# 《小说追踪与推荐平台》开发需求文档

**版本**：1.0  
**日期**：2026-05-26  
**文档状态**：初稿

---

## 1. 项目背景与目标

### 1.1 项目背景
当前小说读者面临信息碎片化、书籍发现困难、更新追踪不便的问题。读者不仅希望阅读内容，更希望在选书阶段快速了解故事的**核心梗概、世界观背景和精彩看点**，同时能方便地关注喜爱的小说，第一时间获取更新动态，并参考他人评价做出阅读决策。

### 1.2 项目目标
构建一个覆盖 Web、App（iOS/Android/Harmony）、微信小程序的多端小说信息聚合平台，为读者提供：
- 精简且高质量的小说**故事概述、背景设定、核心看点**展示。
- 高效的小说**更新追踪**与**个性化信息流**推送。
- 完善的**用户评分与评价体系**，辅助读者决策。
- 流畅的多端统一体验，支持用户注册登录、偏好管理。

---

## 2. 目标用户画像

| 用户类型 | 特征描述 | 核心需求 |
|----------|----------|----------|
| 普通读者 | 网络文学爱好者，希望在选书时快速判断是否符合口味 | 故事梗概、评分、评价、更新提醒 |
| 深度读者 | 有明确追更清单，关注多部作品，不想遗漏更新 | 更新追踪、信息流、书单管理 |
| 社群用户 | 愿意分享和讨论，通过他人评价筛选好书 | 评价互动、点赞、评论回复 |

## 3. 功能性需求

### 3.1 用户系统
**3.1.1 注册与登录**
- 支持手机号 + 短信验证码登录 / 注册。
- 支持微信 OAuth 授权登录（小程序及 App 内优先）。
- 支持游客模式浏览基础内容，但评价、关注等功能需登录。

**3.1.2 个人中心**
- 查看/编辑个人资料（昵称、头像）。
- 我的关注列表：展示已关注的小说，支持取消关注。
- 我的评价列表：可查看、修改或删除自己发表过的评价。
- 我的阅读历史（可选，记录最近浏览的小说详情页）。

### 3.2 小说内容展示
**3.2.1 小说元数据**
每本小说必须具备且清晰展示以下信息：
- 基础信息：书名、作者、封面图、标签（多标签，如“玄幻”“热血”“系统流”）、连载状态（连载中 / 已完结 / 断更 ）。
- **故事概述**：一段 200 - 500 字左右的简洁故事梗概，概括主线剧情。
- **世界观/背景**：描述故事发生的时代、世界架构、力量体系等背景设定。
- **核心看点**：以列表或段落形式突出小说的爽点、创新点、特色人设、感情线等吸引力内容。
- 总字数、最新章节标题与更新时间。

**3.2.2 小说列表与推荐**
- 支持按标签、状态、热度（评分/收藏数）、更新时间进行筛选和排序。
- 展示“热门排行”“新书速递”“编辑推荐”等聚合列表。
- 用户未登录时，展示全站热榜；登录后可在首页看到个性化推荐（后期迭代，初期可暂缺）。

### 3.3 更新追踪与信息流
**3.3.1 小说关注（Follow）**
- 用户在小说详情页点击“关注/追更”，系统记录关注关系。
- 已关注的小说在个人中心列表展示，列表显示每本小说的最新章节标题及更新时间。

**3.3.2 更新动态信息流（Feed）**
- **触发机制**：当后台为某本小说新增或更新了章节信息时，系统自动为所有关注该书的用户生成一条动态。
- **信息流展示**：
  - 登录用户：首页信息流按时间倒序展示自己所关注小说的更新动态。每条动态包含小说封面、书名、最新章节标题、更新时间，点击可跳转至小说详情页或外部阅读源（若有）。
  - 未登录用户：首页展示全站最新更新的小说动态，按时间排序。
- **分页**：支持上拉加载更多，每页 20 条动态。
- **推送通知**（可选，建议后期实现）：关注的小说更新时，通过 App 推送（APNs/FCM）或微信小程序订阅消息通知用户。

### 3.4 评价系统
**3.4.1 发表评价**
- 用户可对任意小说进行评分（1-5 星）并撰写文字评价（最少 10 字）。
- 同一用户对同一本小说只能保留一条评价，新提交的评价会覆盖旧评价。

**3.4.2 评价展示**
- 小说详情页醒目位置展示**综合评分**（平均分，保留一位小数）和评价总条数。
- 评价列表：按时间或热度（点赞数）排序，支持分页。
- 每条评价显示用户头像、昵称、评分星级、内容、发表时间、点赞数。

**3.4.3 评价互动**
- 其他用户可对评价进行“点赞”，点赞数实时更新。
- 用户可删除自己发表的评价，删除后相关点赞数据同步清除。

### 3.5 搜索功能
- 支持按**书名、作者名、标签**进行模糊搜索。
- 搜索结果列表显示小说封面、书名、作者、状态、综合评分，支持高亮关键词。
- 搜索页面提供历史搜索记录（本地存储）和热门搜索推荐。

### 3.6 管理后台（运营人员使用）
- **小说管理**：手动添加、编辑、上架/下架小说，上传封面，录入概述、背景、看点等结构化内容。
- **章节管理**：为指定小说批量导入章节（章节序号、标题、源链接、字数、发布时间），支持爬虫自动抓取（需额外开发）。
- **内容审核**：查看和删除用户评价（违规处理）。
- **数据看板**：展示平台核心指标，如新增用户、关注总数、评价总数、热门小说 Top 10。

## 4. 非功能性需求

| 类别 | 要求 |
|------|------|
| 性能 | - 核心接口（详情、信息流、评价列表）P95 响应时间 < 300ms。<br>- 支持并发 500+ QPS 无感扩容（初期 100 QPS 即可）。 |
| 可用性 | 系统可用性 ≥ 99.5%，关键数据（小说、用户）有定期备份策略。 |
| 安全性 | - 所有接口 HTTPS 加密。<br>- 敏感数据（手机号）脱敏存储。<br>- 防范 SQL 注入、XSS、CSRF 攻击。<br>- JWT 无状态认证，Token 过期时间合理设定。 |
| 可扩展性 | 业务服务与推荐、爬虫服务分离；支持后续引入 Elasticsearch、推荐算法微服务而不影响核心业务。 |
| 兼容性 | Web 端兼容 Chrome、Safari、Edge 最新两个版本；小程序兼容微信最新版本；App 兼容 iOS 15+、Android 10+。 |
| 易用性 | 界面设计遵循移动端规范，操作路径清晰，关键功能（关注、评价）不超过 3 次点击可完成。 |

## 5. 用户界面与交互要点（UI/UX）

- **Web 端**：采用响应式设计，兼顾 PC 和平板。首页顶部导航栏包含“发现”（信息流）、“排行榜”、“搜索”入口；详情页采用三栏或两栏布局，左侧封面，右侧信息，下方评价区。
- **小程序/App**：底部导航栏固定，栏目为“动态”（信息流）、“发现”（榜单）、“书架”（关注列表）、“我的”。小说详情页采用滚动视图，重要操作（关注、写评价）悬浮或置顶。
- **交互反馈**：关注按钮状态变化（未关注/已关注），评分点击有动画效果，信息流下拉刷新、上拉加载。

## 6. 数据需求核心实体

| 实体 | 关键字段 |
|------|----------|
| 用户 (User) | 用户ID、手机号(加密)、微信 OpenID、昵称、头像URL、注册时间、状态 |
| 小说 (Book) | 书籍ID、书名、作者、封面URL、标签 (text[])、故事概述、世界观背景、核心看点、连载状态、总字数、平均评分、评价总数、创建时间、更新时间 |
| 章节 (Chapter) | 章节ID、书籍ID、章节序号、标题、源站链接、字数、发布时间 |
| 用户关注 (Follow) | 关注ID、用户ID、书籍ID、关注时间 |
| 评价 (Review) | 评价ID、用户ID、书籍ID、评分 (1-5)、内容文本、点赞数、创建时间、修改时间 |
| 活动动态 (Feed) | 动态ID、用户ID (接收方)、类型（章节更新）、书籍ID、章节ID、创建时间；用于信息流推送 |

> 数据库选型推荐使用 **PostgreSQL**，利用其数组存储标签，JSONB 存储扩展信息，窗口函数优化信息流查询。

## 7. 技术栈与架构建议（概要）

- **后端**：Spring Boot 4.x + Spring Security + Spring Data JPA + Spring Cache (Redis)
- **数据库**：PostgreSQL 18.4（主库） + 后期增加 Redis 7（缓存/队列） + 可选 Elasticsearch（搜索）
- **消息队列**：RabbitMQ / Kafka（用于更新通知异步解耦，初期可用 Spring Event 模拟）
- **前端多端**：
  - Web：Nuxt 3 (SSR) + Vue 3
  - 小程序：uni-app (Vue 3 版) 或 Taro
  - App：uni-app 
- **基础设施**：MinIO 或阿里云 OSS 存储图片；Nginx 反向代理；GitHub Actions CI/CD

整体采用前后端分离架构，API 遵循 RESTful 风格，使用 JWT 进行无状态认证。

## 9. 附录与备注

- **内容合规**：所有展示的小说信息需遵守知识产权法规，建议与版权方合作或仅提供信息导航，不直接存储侵权正文。作为概述类平台，需在“背景”“看点”等内容上保证原创性描述。
- **爬虫策略**：若采用自动抓取更新，必须遵守目标网站的 robots 协议，控制抓取频率，避免法律风险。
- **接口文档**：所有后端接口需通过 Swagger/OpenAPI 生成文档，方便前端对接。
